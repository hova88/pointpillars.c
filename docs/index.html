A visual systems walkthrough

Turn a point cloud
into upright compute.

PointPillars groups irregular LiDAR returns into vertical columns, learns one vector per live pillar, and spends the rest of inference on a regular bird’s-eye-view CNN. This small C11 runtime makes every shape, byte, and fallback visible.

23.19 MiB model66.786 GMAC12.313 ms warm¹109 boxes²

The network does not learn on raw point order. It first creates a bounded sparse table, then changes representation exactly once for dense convolution.

01 — The idea

Why pillars, not voxels?

A full 3D voxel grid spends memory on empty height slices. PointPillars collapses the entire configured height range into one vertical cell. Every accepted point receives its absolute features, an offset from its pillar mean, and an offset from the pillar center. A tiny PFN max-reduces those rows into one 64-channel vector.

04 — Locality

Sparse on CPU, dense on GPU.

The first convolution is the representation hinge. A CPU can query only live cells through the voxel grid and never create the 64 MiB canvas. A GPU benefits from coalesced dense NCHW traffic, so it clears and scatters that canvas before the CNN.

CPU · direct sparse first layer
65.92 MiB

Live pillar rows are compact; the 1 MiB grid supplies absent-cell zeros. No 64 MiB scatter allocation.

vs
CUDA · materialized NCHW
64 MiB scatter

One asynchronous clear plus coalesced scatter buys regular convolution and persistent device residency.

The right representation depends on the consumer. “Sparse everywhere” and “dense everywhere” are both weaker rules than measuring the boundary.

07 — Memory

The model is small. Activations are not.

HostPPW mmap · 23.19 MiBpillar slot · ≈26.7 MiBtwo-slot prefetch is bounded
features + coords →← raw 14.75 MiB← compact ≈20 KiB
Deviceweights · persistentscatter + stages + aliases206.97 MiB raw cuDNN capacity

Deblock outputs land directly in three adjacent slices of the 384-channel shared input. This is more than a clever pointer trick: it removes a 24 MiB concatenate copy and makes the final layout an ownership decision rather than a postprocessing step.

09 — Reproduce it

Start with the portable path.

  1. 1

    Export the frozen checkpoint

    mkdir -p ckpts
# place pp_multihead_nds5823_updated.pth in ckpts/
make setup-model
make model PYTHON=.venv/bin/python

    PyTorch and PyYAML are offline export dependencies only.

  2. 2

    Build and test C11

    make
make test PYTHON=.venv/bin/python
make portable-test PYTHON=.venv/bin/python

    Linux/WSL uses OpenMP; macOS selects Accelerate/BNNS without Homebrew OpenMP.

  3. 3

    Prepare ten nuScenes sweeps

    make prepare-data PYTHON=.venv/bin/python
frame=$(find /data/nuscenes/pointpillars_10sweep \
  -name '*.bin' -type f | sort | head -1)

    The devkit constructs time lag and transforms sweeps into the keyframe LiDAR frame.

  4. 4

    Measure and validate

    make perf-cpu PERF_FRAME="$frame" PERF_REPS=20
make cuda CUDA_ARCH=sm_89
make cudnn CUDA_ARCH=sm_89
make checkpoint-oracle PERF_FRAME="$frame"

    Use the target GPU architecture; Jetson Orin commonly uses sm_87.

10 — What remains

The next win must beat 12.313 ms end to end.

Potential work includes multi-frame H2D overlap, cuDNN frontend fusion, architecture-specific Jetson kernels, and task-calibrated reduced precision. Each changes scheduling, memory, or numeric behavior; none earns promotion from a microbenchmark alone.

ruleOptimize the measured bottleneck on a named fixture, preserve a strict route, and publish the regression as carefully as the win.
